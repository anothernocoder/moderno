**Onboarding** is the Applications domain capstone (ADR-0005) — a connected set of
[screens](/introduction/blocks/) forming one setup journey: welcome, profile setup, workspace/plan
selection and invite team. Its screens `compose` existing Applications
[blocks](/introduction/blocks/): `profile-setup` composes the Form Layouts block, `plan-select`
composes the Grid Lists block (reusing its title/subtitle/status-badge tile for plan pricing
cards), and `invite-team` composes the List block for the invited-member rows. The CLI's
`composes` resolution is recursive, so pulling the flow walks flow → screen → block automatically.

A **screen** is presentational: it takes a typed wiring contract (`values`/`onNext`, plus
flow-specific navigation callbacks like `onBack`) and owns **no internal state and no router** —
the consumer decides how to store values and navigate. `welcome` needs no block — like the
Checkout flow's `confirmation` screen, it composes only the Button primitive. The **example**
(`Onboarding.example.{ext}`) is a thin runnable assembly that stitches all four screens together
using that framework's own local state (`useState` / `reactive`/`ref` / runes / signals) and a
local `step` variable, carrying the profile values, selected plan and invites between steps — not
the product, just a copy-paste starting point.

## Installation

Pulling the `onboarding` flow copies the four screens it composes (`welcome`, `profile-setup`,
`plan-select`, `invite-team`) **and** the Applications blocks those screens in turn compose
(`form-layouts`, `grid-lists`, `list`) **plus** the example assembly. The CLI resolves `composes`
recursively (flow → screen → block) and dedups by name — primitives are never copied, they stay an
`@moderno/<framework>` npm dependency.

<Tabs syncKey="framework">
<TabItem label="React">
```sh
npx @moderno/registry add onboarding --framework react
```
</TabItem>
<TabItem label="Vue">
```sh
npx @moderno/registry add onboarding --framework vue
```
</TabItem>
<TabItem label="Svelte">
```sh
npx @moderno/registry add onboarding --framework svelte
```
</TabItem>
<TabItem label="Solid">
```sh
npx @moderno/registry add onboarding --framework solid
```
</TabItem>
</Tabs>

This prints every screen and block it pulled in, then copies
`screens/applications/{welcome,profile-setup,plan-select,invite-team}/*.{ext}`,
`{FormLayouts,GridLists,List}.{ext}` and `flows/onboarding/Onboarding.example.{ext}` into `--dest`
(defaults to `./src/blocks`) — screens preserve their nested layout, blocks copy flat to the top of
`--dest`. Pass `--no-example` to copy only the composed screens and blocks:

```sh
npx @moderno/registry add onboarding --framework react --no-example
```

Want just the welcome screen, with no flow context at all? It's addable standalone — and since it
composes no block, nothing else comes along with it:

```sh
npx @moderno/registry add welcome --framework react
```

After copying, install the `@moderno/<framework>` primitive package and
[import the styles](/introduction/installation/). From there, the code — including navigation and
state — is yours.

## Props

### Welcome

| Prop | Type | Description |
|------|------|-------------|
| `onNext` | `() => void` | "Get started" CTA handler |
| `workspaceName` | `string` | If set, appended to the greeting message |
| `title` | `string` | Heading text |
| `description` | `string` | Body text |
| `ctaLabel` | `string` | CTA label |

### ProfileSetup

| Prop | Type | Description |
|------|------|-------------|
| `values` | `Partial<FormLayoutsValues>` | Seed values for the composed FormLayouts block |
| `onNext` | `(values: FormLayoutsValues) => void` | Forwarded from the composed block's `onSave` |
| `onBack` | `() => void` | "Back" link handler, forwarded to the composed block's `onCancel` |
| `title` | `string` | Heading text |
| `description` | `string` | Subheading text |
| `backLabel` | `string` | Label for the back link |

### PlanSelect

| Prop | Type | Description |
|------|------|-------------|
| `plans` | `GridListItem[]` | Plan tiles rendered by the composed GridLists block (falls back to Moderno's own Starter/Pro/Business sample plans) |
| `onNext` | `(plan: GridListItem) => void` | Forwarded from the composed block's `onItemAction` — picking a tile selects the plan and advances immediately |
| `onBack` | `() => void` | "Back" link handler |
| `title` | `string` | Heading text |
| `description` | `string` | Subheading text |
| `backLabel` | `string` | Label for the back link |

### InviteTeam

| Prop | Type | Description |
|------|------|-------------|
| `values` | `InviteTeamValues` | Controlled value of the "add teammate" email field |
| `onChange` | `(values: InviteTeamValues) => void` | Fired as the email field changes |
| `onInvite` | `(values: InviteTeamValues) => void` | Fired when the invite form is submitted |
| `invites` | `ListItem[]` | Pending/accepted invites rendered by the composed List block |
| `onRemoveInvite` | `(item: ListItem) => void` | Forwarded from the composed block's `onItemAction` |
| `onNext` | `() => void` | "Finish" CTA handler |
| `onBack` | `() => void` | "Back" link handler |
| `title` | `string` | Heading text |
| `description` | `string` | Subheading text |
| `emailPlaceholder` | `string` | Placeholder for the email field |
| `inviteLabel` | `string` | Label for the invite button |
| `nextLabel` | `string` | Label for the finish CTA |
| `backLabel` | `string` | Label for the back link |

No screen carries internal state or defaults for profile/plan/invite values — that's the example
assembly's job (or your own code). `welcome` composes only the Button primitive; `profile-setup`,
`plan-select` and `invite-team` compose their respective Applications block plus the
Input/Button primitives for their own chrome. All use the `--md-*` tokens; no legacy utility
classes.

## Usage

```tsx
import { useState } from 'react'
import { Welcome } from './screens/applications/welcome/Welcome'

function App() {
  const [step, setStep] = useState('welcome')

  return (
    <Welcome
      onNext={() => {
        setStep('profile')
        router.push('/onboarding/profile')
      }}
    />
  )
}
```

Wiring in your own router/state is the point: no screen ever assumes one exists. See
`Onboarding.example.tsx` (and its `.vue`/`.svelte`/`.solid.tsx` siblings) for the full journey — a
local `step` union across all four screens, the profile values carried from `profile-setup`
forward, the selected plan carried from `plan-select` forward, and the growing invites list built
up in `invite-team`.
