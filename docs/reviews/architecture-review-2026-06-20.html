Architecture review

Moderno — deepening opportunities

Token-driven, framework-agnostic design system · React · Vue · Svelte · Solid · 20 Jun 2026

module seam leakage / duplication deep module

Two modules already pull their weight: @moderno/class-contract and @moderno/styles are deep and tested — a lot of behaviour (every legal md-* name, the CSS contract check) behind a small interface. The friction lives entirely in the styled layer — the per-framework code that wires Zag's prop-getters to the class contract. The same binding recipe is hand-written four times, with no module owning it, and it is the one part of the system with no test surface.

Strong in-process

1 · Collapse the part-context triad into one helper

Before — shallow, repeated per primitive × per framework

flowchart TB
  subgraph one[" "]
    direction TB
    T1["Tabs: createContext + useTabs + throw"]
    A1["Accordion: createContext + useAccordion + throw"]
    S1["Select: createContext + useSelect + throw"]
  end
  T1 -.dup.-> A1
  A1 -.dup.-> S1
  classDef dup stroke:#dc2626,stroke-width:2px,fill:#fef2f2;
  class T1,A1,S1 dup

Same create / read / "must be used inside Root" throw, re-typed for each compositional primitive — and again in each of the four framework packages.

After — one deep helper per framework

createPartContext(name)
▼ one interface
Tabs
Accordion
Select

Returns { Provider, usePart }; the throw message is generated from name. Grows for free as ADR-0003's Date picker, Carousel, Sheet land.

Problem. The context create/read/throw triad is re-implemented for every compositional primitive — shallow code with no single home.

Solution. One createPartContext(name) per framework; primitives call it instead of restating the triad.

  • locality: the throw + null-guard live in one module
  • leverage: one interface, N compositional primitives
  • interface shrinks; implementation absorbs the triad
  • deletion test: passes — complexity reappears across callers
Strong local-substitutable

2 · Give the styled layer a test surface at its interface

Interface = test surface — where it is asserted today

flowchart LR
  CC["class-contract\n(pure)"] --> T1{{"vitest ✓"}}
  ST["styles\n(pure)"] --> T2{{"vitest ✓"}}
  R["react styled layer"] --> X1["tsc only — DOM unasserted"]
  V["vue styled layer"] --> X2["vue-tsc only"]
  S["svelte styled layer"] --> X3["svelte-check only"]
  SO["solid styled layer"] --> X4["tsc only"]
  classDef ok stroke:#059669,stroke-width:2px,fill:#ecfdf5;
  classDef gap stroke:#dc2626,stroke-width:2px,fill:#fef2f2;
  class T1,T2 ok
  class X1,X2,X3,X4 gap

The pure modules are tested. The hand-written Zag glue — the riskiest code, repeated four times — is checked only by the type-checker. Its interface is the rendered DOM: right class on right element, ARIA wired from Zag, the context throw firing. None of that is observable through tsc.

Before

class-contract — tested
styles — tested
styled layer × 4 — type-checked only

After

render Tabs → assert anatomy + classes
render Accordion → assert ARIA from Zag
render part outside Root → assert throw

Problem. The styled layer's interface (rendered DOM + a11y) is never asserted; bugs in the ×4 glue surface only in the docs app or in a consumer.

Solution. Add a Testing-Library render test per framework that asserts each primitive's anatomy through its interface — classes on the right elements, Zag's ARIA, the context throw.

  • the interface becomes the test surface
  • locality: anatomy regressions caught at one seam
  • de-risks every later deepening (1 & 3)
  • tests survive internal refactors — assert DOM, not state
Worth exploring in-process

3 · Lift the anatomy recipe into a single declarative spec

Before — recipe re-encoded as prose, ×4

flowchart TB
  REC(["Tabs anatomy recipe:\nTrigger = button + getTriggerProps + md-tabs-trigger"])
  REC -.-> RR["react/tabs.tsx"]
  REC -.-> VV["vue/Tabs*.vue"]
  REC -.-> SS["svelte/Tabs*.svelte"]
  REC -.-> OO["solid/tabs.tsx"]
  classDef ghost stroke-dasharray:5 5,fill:#fff,stroke:#94a3b8;
  classDef dup stroke:#dc2626,fill:#fef2f2;
  class REC ghost
  class RR,VV,SS,OO dup

The recipe is a real module with no file. parts.* owns the class names; element tag + prop-getter binding is still copied by hand.

After — spec owns the binding; renderers interpret it

flowchart TB
  SPEC["anatomy spec\n(part → tag · getter · class)"]
  SPEC --> RA["react leaf renderer"]
  SPEC --> VA["vue leaf renderer"]
  SPEC --> SA["svelte leaf renderer"]
  SPEC --> OA["solid leaf renderer"]
  classDef deep fill:#0f172a,color:#fff,stroke:#0f172a;
  class SPEC deep

Leaf parts genericize cleanly. Root stays hand-written — its controlled-prop wiring differs per machine and per framework reactivity model.

Scope guard: deepen the leaf parts only. A fully generic Root would leak the four reactivity models (Svelte runes, Solid getters, React/Vue context) through its interface — that is where this turns speculative.

Problem. The part→tag→getter→class binding is shallow and duplicated ×4 with no owning module; adding a primitive means four parallel edits.

Solution. Extend class-contract's parts into an anatomy spec naming each part's element + Zag getter; per-framework leaf renderers interpret it.

  • leverage: one spec, four renderers, N primitives
  • locality: anatomy changes land in one module
  • pairs with candidate 2 — one conformance test per spec
  • extends the "one core, four adapters" shape already in use
Speculative in-process

4 · Deepen overlay glue (Portal + open guard)

⚠ Contradicts ADR-0002. The ADR explicitly accepts hand-written composition glue (Portal, presence/exit, open guard) per framework × 4 for Dialog, Tooltip, Toast, Popover, Select as the cost of dropping Ark. Surfaced only because the duplication is concrete — not recommended for action without reopening the ADR. 
flowchart LR
  D["Dialog"] -.same open+portal.-> P["Popover"]
  P -.same open+portal.-> Tt["Tooltip"]
  Tt -.same open+portal.-> Sel["Select.Content"]
  classDef dup stroke:#dc2626,stroke-width:2px,fill:#fef2f2;

If reopened: a per-framework <Overlay> that owns the api.open guard + Portal wiring. Two real adapters justify it (prod Portal vs test inline), but ADR-0002 judged the glue cheaper than the abstraction. Revisit only if a presence/exit-animation bug shows up in three+ overlays at once.

Top recommendation

Start with candidate 2 — a test surface for the styled layer

The styled layer is the only untested module and the only one written four times — the highest-risk code with the weakest seam. Asserting each primitive through its interface (rendered DOM + Zag's ARIA) turns that interface into a test surface and de-risks the two deepenings that follow: the part-context helper (1) and the anatomy spec (3) both rewrite this code, and conformance tests are what let you collapse it without fear. Land tests first, then deepen — in that order.
