**Auth** is Moderno's first [flow](/introduction/blocks/) (ADR-0005): a connected set of
[screens](/introduction/blocks/) forming one user journey. It ships **five presentational
screens** — sign-in, sign-up, forgot-password, reset-password and verify — plus a thin example
assembly that wires them together with local state: a sign-in↔sign-up toggle, a "forgot
password" branch that verifies a code before letting the user pick a new password, and an email
address carried forward from sign-up/forgot-password into the verify screen.

A **screen** is presentational: it takes a typed wiring contract (`values`, `onChange`,
`onSubmit`, plus flow-specific navigation callbacks like `onSignUp`/`onSignIn`/`onBack`) and owns
**no internal state and no router** — the consumer decides how to store values and navigate. The
**example** (`Auth.example.{ext}`) is a thin runnable assembly that stitches all five screens
together using that framework's own local state (`useState` / `reactive` / runes / signals) and a
local `step` variable — not the product, just a copy-paste starting point. The live demo below
drives the example through a single serializable `step` string prop (the Astro-island boundary
serializes to JSON, so only plain strings cross it) — everything else is the example's own state.

## Installation

Pulling the `auth` flow copies the five screens it composes (`sign-in`, `sign-up`,
`forgot-password`, `reset-password`, `verify`) **plus** the example assembly. The CLI resolves
`composes` recursively and dedups by name — pull any single screen alone if that's all you want.

<Tabs syncKey="framework">
<TabItem label="React">
```sh
npx @moderno/registry add auth --framework react
```
</TabItem>
<TabItem label="Vue">
```sh
npx @moderno/registry add auth --framework vue
```
</TabItem>
<TabItem label="Svelte">
```sh
npx @moderno/registry add auth --framework svelte
```
</TabItem>
<TabItem label="Solid">
```sh
npx @moderno/registry add auth --framework solid
```
</TabItem>
</Tabs>

This copies `screens/applications/{sign-in,sign-up,forgot-password,reset-password,verify}/*.{ext}`
and `flows/auth/Auth.example.{ext}` into `--dest` (defaults to `./src/blocks`), preserving that
relative layout so the example's imports of each screen keep working. Pass `--no-example` to copy
only the composed screens:

```sh
npx @moderno/registry add auth --framework react --no-example
```

Want just one screen, no flow context at all? Add it directly:

```sh
npx @moderno/registry add sign-up --framework react
```

After copying, install the `@moderno/<framework>` primitive package and
[import the styles](/introduction/installation/). From there, the code — including navigation
and state — is yours.

|------|------|-------------|
| `values` | `{ email: string; password: string; remember: boolean }` | Controlled field values |
| `onChange` | `(values) => void` | Called with the next `values` on every field change |
| `onSubmit` | `(values) => void` | Called with `values` on form submit |
| `onForgotPassword` | `() => void` | "Forgot password?" link handler |

No screen carries internal state or defaults for `values` — the example assembly (or your own
code) owns that. All five compose the Input, Button and (sign-in/sign-up) Checkbox/Divider
primitives and the `--md-*` tokens; no legacy utility classes.

## Usage

```tsx
import { useState } from 'react'
import { SignIn, type SignInValues } from './screens/applications/sign-in/SignIn'

function App() {
  const [values, setValues] = useState<SignInValues>({ email: '', password: '', remember: false })

  return (
    <SignIn
      values={values}
      onChange={setValues}
      onSubmit={(v) => fetch('/api/sign-in', { method: 'POST', body: JSON.stringify(v) })}
      onForgotPassword={() => router.push('/forgot-password')}
      onSignUp={() => router.push('/sign-up')}
    />
  )
}
```

Wiring in your own router/state is the point: no screen ever assumes one exists. See
`Auth.example.tsx` (and its `.vue`/`.svelte`/`.solid.tsx` siblings) for the full journey — a local
`step` union across all five screens, an email carried from sign-up/forgot-password into verify,
and a `verifyPurpose` flag that decides whether a successful code returns you to sign-in (new
account) or forwards you to reset-password (recovery).
