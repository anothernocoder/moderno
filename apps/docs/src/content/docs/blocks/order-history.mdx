**Order History** is an ecommerce [block](/introduction/blocks/): a list of a customer's
past orders — each row showing the order number, date, a status pill, and total — that
links into order details. It composes the `Card` and `Badge` primitives plus the
`--md-*` tokens. You copy it into your repo with the CLI and edit it freely.

Like [Order Summaries](/blocks/order-summaries/), this block is purely presentational:
each order's `href` is a plain anchor link, with no routing logic baked in — wire it up
to your own router or leave it as a normal link to a server-rendered details page.

## Props

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `heading` | `string` | `'Historial de pedidos'` | Heading above the list (omit to hide it) |
| `orders` | `OrderHistoryOrder[]` | 4 sample orders | Past orders — each `{ id, orderNumber, date, status, total, href }` |
| `emptyMessage` | `string` | `'Todavía no tienes pedidos.'` | Message shown when `orders` is empty |
| `detailsLabel` | `string` | `'Ver detalles'` | Text for each order's details link |

`status` is one of `'delivered' | 'shipping' | 'processing' | 'cancelled'`, mapped
internally to a Spanish label and a `Badge` variant:

| Status | Label | Badge variant |
|--------|-------|---------------|
| `delivered` | Entregado | `success` |
| `shipping` | En camino | `info` |
| `processing` | Procesando | `warning` |
| `cancelled` | Cancelado | `error` |

## Usage

```tsx
import { OrderHistory } from './blocks/OrderHistory'

<OrderHistory
  orders={[
    { id: 'ord-1', orderNumber: '1042', date: 'July 15, 2026', status: 'delivered', total: 458000, href: '/orders/1042' },
    { id: 'ord-2', orderNumber: '1039', date: 'July 2, 2026', status: 'shipping', total: 210000, href: '/orders/1039' },
  ]}
/>
```

The props API is identical across all four frameworks. Each order's `total` is a plain
`number` (formatted internally with `toLocaleString`), and `href` is rendered as a plain
`<a>` tag — no client-side routing logic is included, so plug in your framework's router
`Link` component instead if you need one. No new primitives, no bespoke interaction
logic — just the existing `Card` and `Badge` primitives composed with layout.
