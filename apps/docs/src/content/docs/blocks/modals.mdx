**Modals** is an application [block](/introduction/blocks/): an account-actions panel — a
list of rows, each pairing a label/description with a trigger that opens a centered
[`Dialog`](/components/dialog/). The sample data shows three common patterns: an
informational "edit profile" dialog, an acknowledgement "export data" dialog, and a
destructive-toned "delete account" warning. It composes the `Badge` and `Dialog` primitives
and the `--md-*` tokens.

`Dialog` is a **closed-prop** primitive (ADR-0003): its trigger, title, description, and
single close action are all it exposes — there's no bespoke interaction logic layered on top
here, just multiple `Dialog` instances arranged in a settings-panel layout.

## Props

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `title` | `string` | `'Gestión de cuenta'` | Panel heading |
| `description` | `string` | Sample copy | Panel subheading (hidden if empty) |
| `actions` | `ModalAction[]` | 3 sample actions | The rows to render, each backed by its own `Dialog` |

`ModalAction` shape:

| Field | Type | Description |
|-------|------|--------------|
| `id` | `string` | Unique row id / React key |
| `label` | `string` | Row label |
| `description` | `string` | Row description |
| `tone` | `'default' \| 'danger'` | `'danger'` tints the label and adds an "Irreversible" `Badge` |
| `triggerLabel` | `string` | `Dialog`'s `triggerLabel` |
| `dialogTitle` | `string` | `Dialog`'s `title` |
| `dialogDescription` | `string` | `Dialog`'s `description` |
| `closeLabel` | `string` | `Dialog`'s `closeLabel` |

## Usage

```tsx
import { Modals } from './blocks/Modals'

<Modals
  title="Account"
  actions={[
    {
      id: 'edit-profile',
      label: 'Edit profile',
      description: 'Update your name, email, and contact details.',
      triggerLabel: 'Edit profile',
      dialogTitle: 'Edit profile',
      dialogDescription: 'Update your name, email, and contact details from the settings form.',
      closeLabel: 'Done',
    },
    {
      id: 'delete-account',
      label: 'Delete account',
      description: 'This action is permanent and cannot be undone.',
      tone: 'danger',
      triggerLabel: 'Delete account',
      dialogTitle: 'Delete your account?',
      dialogDescription: 'All your data, orders, and history will be permanently deleted. This cannot be undone.',
      closeLabel: 'Cancel',
    },
  ]}
/>
```

The props API is identical across all four frameworks. Since `Dialog` only exposes a single
close trigger (no `onClose`/second action button), a "danger" row's dialog can warn and let
the user back out — it can't itself perform the destructive action. If you need a real
confirm step, that means composing a different flow than a closed-prop `Dialog` alone can
give you: for example a two-step trigger (click "Delete account" to reveal an inline confirm
button, `Dialog` still handling the warning copy), which is exactly the kind of layout
decision this block leaves to you once you've copied it into your repo.
