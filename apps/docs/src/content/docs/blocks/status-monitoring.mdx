**Status monitoring** is an application [block](/introduction/blocks/): a service health panel
with an overall status banner and a list of monitored services, each showing a status
`Indicator` dot, a `Badge`, and optional uptime / last-incident text. It composes the `Badge`,
`Divider`, and `Indicator` primitives and the `--md-*` tokens.

## Props

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `title` | `string` | `'Estado del sistema'` | Panel header title |
| `services` | `ServiceItem[]` | 4 sample services | Monitored services to render |

### `ServiceItem`

| Field | Type | Description |
|-------|------|--------------|
| `id` | `string` | Unique service id, used as the list key |
| `name` | `string` | Service name |
| `status` | `'operational' \| 'degraded' \| 'down'` | Health state — drives the `Indicator`/`Badge` color (`operational` → success, `degraded` → warning, `down` → error) and adds a pulse to the indicator when `down` |
| `statusLabel` | `string` | Status text shown in the `Badge` |
| `uptime` | `string` | Pre-formatted uptime percentage (hidden if omitted) |
| `lastIncident` | `string` | Last-incident summary text (hidden if omitted) |

The overall status banner in the header is derived automatically from the worst status found
across `services` (`down` > `degraded` > `operational`).

## Usage

```tsx
import { StatusMonitoring } from './blocks/StatusMonitoring'

<StatusMonitoring
  title="System status"
  services={[
    { id: 'api', name: 'API', status: 'operational', statusLabel: 'Operational', uptime: '99.98%', lastIncident: 'No incidents in 90 days' },
    { id: 'payments', name: 'Payments', status: 'degraded', statusLabel: 'Degraded', uptime: '98.40%', lastIncident: 'Incident 2 hours ago' },
  ]}
/>
```

The props API is identical across all four frameworks.
